# AI 行程任务台设计

日期：2026-07-06
状态：待评审

## 背景

宠迹 AI 现有 AI 行程模块已经具备可运行基础：`PlannerPage` 提供推荐行程、热门目的地和输入入口；`ChatView` 提供对话、Mock/真实 AI 调用、结构化地点卡片和地图跳转；`mockAiEngine` 能按城市、天数、餐厅、室内等关键词输出 `prose`、`itinerary`、`risks`、`checklist`；地点数据已有宠物规则、验证人数、最近验证时间和评分；全局 store 已有宠物档案、聊天持久化、地图地点、社区 Feed 与验证回写。

当前问题不是“没有 AI 行程”，而是路演时看起来仍像普通聊天 Demo。评委需要读完输入、等待回复、点进地图后，才可能理解“宠物友好数据 + AI + 地图 + 社区验证”的闭环。下一轮应该把 AI 行程做成产品卖点入口，让评委在 10 秒内看懂：宠迹 AI 不是聊天机器人，而是能组织携宠出行决策流程的任务台。

## 已确认方向

本轮目标优先级是路演和比赛冲击力。用户选择的核心表达是：AI 行程要体现“宠物友好数据 + AI + 社区验证”的产品闭环。

推荐采用组合方案：

1. `PlannerPage` 升级为“AI 行程任务台”首页。
2. `ChatView` 保留作为生成和追问入口，但生成后的结果展示升级为计划工作台。
3. 不单独做故事介绍页。故事感只用于空状态、示例任务和首屏价值表达。

## 目标

1. 评委打开 AI 行程页后，首屏即可理解产品闭环：档案授权、地点规则、AI 规划、地图执行、社区验证。
2. 复用现有代码和数据，不推翻现有聊天、Mock AI、地图跳转、社区验证能力。
3. 让 AI 结果从“地点卡片列表”升级为“可执行计划”：时间线、规则依据、风险、清单、地图动作、验证动作。
4. 保持 Demo 稳定性：离线 Mock 路径仍然可用，真实 API 失败仍可回退 Mock。
5. 明确宠物档案隐私边界：只有用户开启授权时，宠物档案才进入 AI 上下文。

## 非目标

1. 本轮不做真实后端、账号、云端同步、商家后台或审核系统。
2. 不重写地图页、社区页或宠物档案页，只从 AI 行程页串联已有能力。
3. 不引入复杂路线导航、真实距离计算、实时天气、真实 POI 搜索。
4. 不把 AI 首页做成纯营销页；它必须仍然是可输入、可生成、可继续追问的工具入口。

## 设计概览

AI 行程模块分为两个状态。

任务台首页：由现有 `PlannerPage` 承担。它负责“第一眼卖点”和快速发起演示。首屏展示一个携宠出行任务台，而不是普通推荐列表。

生成后工作台：由现有 `ChatView` 承担。它负责承接用户输入、AI 回复、多轮追问、地图跳转和验证动作。AI 回复不再只渲染地点卡片，而是渲染一份结构化计划。

这两个状态共享现有数据：`pets`、`city`、`settings`、`chat`、`PLACES`、`PLACE_MAP`、`verifyPlace`、`feeds`。首版不需要新路由，仍沿用 `/ai` 和 `/ai/chat`。

## 任务台首页

`PlannerPage` 的首屏应该由以下区域组成。

### 1. 顶部价值区

展示一句明确价值主张：

“让 AI 把宠物档案、地点规则和社区验证变成一份可执行行程。”

旁边保留当前宠物头像和城市信息。这里不再只问“今天想带豆豆去哪里玩”，而是直接说明 AI 在做什么。

### 2. 闭环信号板

用 4 个紧凑信息块展示当前计划会综合的信号：

1. 宠物档案：宠物名、体型、性格；如果未授权，展示“未授权，点击启用”。
2. 地点规则：候选地点数量、室内可进数量、大型犬友好数量。
3. 风险约束：高温、雨天、室内限制、服务费等 Demo 级约束。
4. 社区验证：最近验证时间、验证人数、规则可信度。

这些信息必须来自现有 `pets`、`PLACES`、`places`、`feeds` 或轻量派生计算，不新增远程依赖。

### 3. 路演示例任务

展示一条主推示例，例如：

“周六带豆豆在上海玩一天，午餐要室内可进，下午避开高温。”

示例卡片里应直接预览计划骨架：上午释放精力、午餐室内、下午低强度、傍晚验证打卡。点击后清空旧聊天并跳转 `/ai/chat?q=...`，复用现有 pendingText 自动发送逻辑。

### 4. 快速输入

保留底部输入入口，但文案从“告诉 AI 你想去哪里”升级为“描述你的宠物、城市和出行限制”。输入点击进入 `ChatView`，沿用现有路由。

### 5. 场景快捷按钮

保留推荐入口，但从地点/路线列表改为更有产品感的场景：

1. 大型犬友好一日游
2. 雨天室内备选
3. 两天一夜含住宿
4. 周末避开人流

这些按钮只负责生成高质量 prompt，不在首页模拟完整 AI。

## 生成后工作台

`ChatView` 继续承担聊天交互，但 Assistant 回复应包含更完整的结构化视图。

### 1. 计划摘要

在 AI 自然语言回复下方展示计划摘要：

- 城市和天数
- 使用的宠物档案状态：已授权或未授权
- 当前计划可信度：来自地点规则和社区验证的综合展示，不需要复杂算法
- Mock/真实 AI 来源提示

### 2. 时间线

把 `itinerary` 渲染为时间线，而不是独立地点卡片列表。每一步展示：

- 时间
- 行程标签
- 地点名称
- 宠物规则摘要
- 为什么选它
- 操作：查看地图、替换、收藏

首版“替换”可以先作为追问 prompt：点击后把“把这一步换成室内可进/人少/大型犬友好”填入输入框，不需要实现复杂局部重算。

### 3. 风险与备选

`risks` 不应只是折叠文本。首版按风险类型展示：

- 规则风险：大型犬限制、室内限制、服务费
- 环境风险：高温、雨天、人流
- 执行风险：预约、交通、牵引要求

如果数据不足，Mock AI 生成可解释的 Demo 风险即可。真实 API 响应缺少分类时，前端可用字符串降级展示。

### 4. 行前清单

`checklist` 继续展示，但与宠物档案产生关联：如果宠物体型大，突出牵引和饮水；如果多日行程，突出住宿、证件和备用粮。

### 5. 闭环动作

生成后的计划必须至少露出三个动作：

1. 查看地图：跳转 `/map?place=<id>`，复用现有逻辑。
2. 标记已验证：调用现有 `verifyPlace`，产生本地社区 Feed。
3. 继续调整：把追问写入输入框，保持多轮对话。

这样 AI 行程页能直接演示“计划 -> 地图 -> 验证 -> 社区回写”。

## AI 响应结构

现有 `AiReply` 保持兼容：

```ts
interface AiReply {
  prose: string
  itinerary: AiItineraryStep[]
  risks: string[]
  checklist: string[]
}
```

本轮可以在兼容基础上扩展可选字段，不要求一次迁移所有真实 API 响应：

```ts
interface AiReply {
  prose: string
  itinerary: AiItineraryStep[]
  risks: string[]
  checklist: string[]
  summary?: {
    title: string
    city?: string
    days?: number
    confidenceLabel?: string
    source?: 'mock' | 'api' | 'fallback'
  }
  riskSections?: Array<{
    type: 'rule' | 'environment' | 'execution'
    title: string
    items: string[]
  }>
}
```

`AiItineraryStep` 可以追加可选字段：

```ts
interface AiItineraryStep {
  time?: string
  placeId?: string
  label: string
  ruleBrief?: string
  action?: string
  reason?: string
  verifyHint?: string
  alternatives?: Array<{ placeId: string; reason: string }>
}
```

旧字段继续可用。前端渲染必须能处理缺少新增字段的响应。

## Mock AI 增强

`mockAiEngine` 的职责从“凑一个地点序列”升级为“稳定生成路演级结构化计划”。它仍然只基于现有 `PLACES` 数据，不接网络。

首版规则：

1. 识别城市、天数、室内、住宿、雨天、周末、大型犬等意图。
2. 根据宠物体型过滤或标注地点规则风险。
3. 每一步生成 `reason`，解释为什么选这个地点。
4. 每一步生成 `verifyHint`，提示用户到现场要确认什么。
5. 风险按 `rule`、`environment`、`execution` 三类输出。
6. 多轮“换/调整/室内/人少/两天”继续可用，至少能稳定改晚餐、住宿和天数。

## 宠物档案隐私

当前 UI 和数据流必须清楚区分“有宠物档案”和“授权给 AI”。任务台首页应提供显式授权控件，状态来自 `showPetInChat`。

AI 请求规则：

1. `showPetInChat=false` 时，不向 Mock AI 或真实 API 传入 `petContext`。
2. `showPetInChat=true` 时，传入用户选择的宠物摘要。
3. UI 文案必须让用户理解：档案默认本地私密，开启后仅用于本次/当前 AI 规划上下文。

如果本轮发现现有代码默认传入宠物档案，应作为实施时的必修修正。

## 组件边界

建议新增或调整这些 AI 页面内组件，不引入全局大重构：

1. `MissionControlHero`：任务台顶部价值区。
2. `PlanningSignals`：闭环信号板。
3. `DemoScenarioCard`：路演示例任务。
4. `PlanSummaryCard`：生成后计划摘要。
5. `ItineraryTimeline`：结构化时间线，替代单纯地点卡片列表。
6. `RiskPanel`：风险分类展示。
7. `ChecklistPanel`：行前清单。
8. `PlanActions`：地图、验证、追问动作。

现有 `PlaceCard` 可继续复用或被 `ItineraryTimeline` 包装。不要把所有 UI 继续堆在 `ChatView.tsx`，避免该文件继续膨胀。

## 交互流程

### 路演默认流程

1. 评委打开 `/ai`。
2. 看到任务台首页：宠物、地点规则、风险、社区验证四类信号。
3. 点击主推示例任务。
4. App 跳转 `/ai/chat?q=...` 并自动生成计划。
5. AI 回复展示摘要、时间线、风险、清单。
6. 点击第一步“查看地图”，跳到地图并高亮地点。
7. 返回或继续操作“标记已验证”，生成社区 Feed。

### 用户自由输入流程

1. 用户点击输入框进入 `/ai/chat`。
2. 输入自然语言出行需求。
3. AI 使用已授权上下文和当前城市候选地点生成计划。
4. 用户继续追问调整，系统保留对话上下文。

## 错误和降级

1. 真实 API 失败：保留现有 Toast，说明已切 Mock，并在计划摘要标记“Mock fallback”。
2. AI 返回非法 JSON：不破坏聊天渲染，显示自然语言错误提示，并提供“用 Mock 重新生成”入口。
3. 地点 ID 找不到：时间线显示地点名或“推荐地点”，地图动作禁用。
4. 宠物档案为空：任务台提示“添加宠物可获得更贴合计划”，但仍允许无档案规划。
5. 地图瓦片失败：地图页已有降级，本轮不扩展；AI 侧只负责跳转和高亮参数。

## 测试和验收

### 自动化验收

1. 打开 `/ai`，能看到任务台价值文案、闭环信号板、主推示例任务。
2. 点击主推示例后，能跳转 `/ai/chat?q=...` 并生成结构化计划。
3. 计划中至少出现时间线、风险、清单和“查看地图”动作。
4. 点击“查看地图”后，URL 包含 `/map?place=<id>`。
5. `showPetInChat=false` 时，Mock AI 输入上下文不含宠物档案字段。
6. `showPetInChat=true` 时，Mock AI 可收到宠物摘要。
7. 真实 API 失败时仍回退 Mock，聊天不白屏。

### 人工验收

1. 评委视角 10 秒内能看懂 AI 行程页不是普通聊天。
2. 首屏能看出地图、宠物档案、地点规则、社区验证共同参与计划。
3. 生成结果看起来像可执行行程，而不是地点列表。
4. 移动端 375px 和桌面端 1440px 不出现遮挡、横向滚动或按钮文案溢出。

## 风险

1. 首屏信息过多会变成仪表盘噪音。解决方式：只展示 4 个信号，每个信号一句话。
2. 结果页过重会影响聊天轻快感。解决方式：首条计划用结构化卡片，追问回复可简化。
3. Mock AI 过度包装会显假。解决方式：所有理由都绑定已有地点规则或社区验证字段。
4. 宠物档案授权边界若处理不清，会削弱隐私卖点。实施时必须先修正默认传宠物上下文的问题。

## 下一步

本 spec 评审通过后，进入实施计划阶段。实施计划应按风险拆为：

1. 隐私授权和 AI 上下文修正。
2. 任务台首页 UI。
3. Mock AI 输出增强。
4. 生成后工作台组件拆分。
5. E2E 和回归测试更新。
