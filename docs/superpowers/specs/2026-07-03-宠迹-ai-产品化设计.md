# 宠迹 AI 产品化设计

日期：2026-07-03
状态：已确认，作为后续迭代基准

## 背景

宠迹 AI 当前不是一个空白原型，而是一个已经比较完整的演示版。项目已经具备 Vite + React + TypeScript 前端、路由级懒加载、Tailwind UI、Leaflet 地图、Mock 城市数据、Mock AI 行程规划、OpenAI 兼容调用、LocalStorage 持久化、PWA service worker、GitHub Pages 部署，以及一条自定义回归脚本。

下一阶段不应该继续堆很多无关页面，而应该做产品化打磨：先让工程质量可控，再拆清状态边界，然后把 AI 和数据能力向真实 MVP 架构推进，最后强化真实用户闭环。

## 目标

1. 让现有 demo 具备可持续迭代的工程基线，避免改一处坏一片。
2. 把 demo 阶段形成的全局状态拆成更清晰的业务域，并稳定持久化契约。
3. 在保留离线演示能力的前提下，让 AI 调用和数据模型逐步接近真实 MVP。
4. 强化核心产品闭环：规划携宠行程、查看地点、验证规则、贡献社区反馈，并在用户主动授权时把宠物档案提供给 AI。

## 非目标

1. 本轮不做完整商家后台、支付、会员或交易系统。
2. 不重写整个前端技术栈。
3. 不移除离线 Mock 演示路径，它仍然对路演和容灾有价值。
4. 在数据模型、验证模型和内容流程可靠前，不盲目扩展大量城市。

## 推荐路线

### 阶段 1：工程基线

目的：让现有代码变得安全、可验证、可持续修改。

范围：
- 增加 CI 质量门，覆盖 pull request 和 main 分支 push：安装依赖、lint、test、build。
- 修复当前 lint warning，让后续 warning 真正有信号价值。
- 把现有自定义回归脚本显式纳入质量命令；后续再判断是继续作为 smoke coverage，还是逐步迁入标准测试框架。
- 增加浏览器级烟测，覆盖关键路径：AI 规划、地图跳转、地点验证、社区反馈、设置持久化、宠物档案隐私边界。
- 记录构建产物体积，并设置初始 bundle budget。

验收：
- `npm run lint`、`npm test`、`npm run build` 在本地和 CI 都通过。
- CI 能在质量检查失败时阻断合并。
- 至少有一条端到端烟测覆盖产品主闭环。
- 当前已知 warning 被修复或明确记录为有意保留。

### 阶段 2：架构拆分

目的：降低耦合，让后续功能更容易理解、测试和演进。

范围：
- 将当前全局 store 拆为 settings、pets、places、community、chat、ui 等业务域。
- 将 seed data 和 mock feed 生成逻辑移出运行时 store。
- 增加明确的 LocalStorage schema version 和 migration 函数。
- 缩窄 selector，让组件只订阅自己真正需要的状态。
- 将 `verifyPlace` 等业务 action 移到可单独测试的领域模块。
- 拆分后复查 bundle 输出，重点关注当前偏大的共享 store chunk。

验收：
- Hydration 和 persistence 对已有 LocalStorage 数据保持兼容。
- 现有回归测试继续通过。
- 每个 store 模块足够小，能独立理解。
- 功能页面不再通过全局 store 间接引入无关 demo 数据。

### 阶段 3：AI 与数据产品化

目的：让 AI 规划和地点数据对真实用户更可信。

范围：
- 引入 AI gateway 边界。本地 demo 模式仍可使用 Mock AI 或用户自备 key；产品模式下，AI 请求应经过后端或 edge proxy，而不是在浏览器直接暴露 provider 细节。
- 渲染前用严格 itinerary schema 校验 AI 响应。
- 拆分 prompt 构造、provider transport、响应归一化和 UI 渲染。
- 增加更明确的错误分类：缺 key、认证失败、限流、超时、非法 JSON、provider 不可用、已启用 fallback。
- 扩展地点数据字段：来源、最近确认时间、可信度、贡献人数、审核状态。
- 将 verification event 作为一等记录，而不是只改地点计数器。

验收：
- UI 能说明当前计划来自真实 AI 还是 Mock fallback。
- 非法 AI 响应不会破坏聊天渲染。
- 地点验证历史可以从记录审计，而不只依赖计数器推断。
- 宠物档案只在用户明确授权后进入 AI 上下文。

### 阶段 4：真实用户闭环

目的：聚焦最强产品闭环，而不是继续横向扩功能。

范围：
- 优化从 AI 计划到地点详情再到验证的路径。
- 如能直接支持复用和留存，再增加收藏地点或保存计划。
- 将与地点验证绑定的社区帖子结构化：是否到访、规则是否变化、体型兼容、是否可进室内、费用、自由备注。
- 将桌面端从手机模拟器升级为真实响应式 Web 布局，同时保留演示需要的模拟器入口。
- 增加隐私友好且可关闭的基础 analytics hook；如果暂不接受 analytics，则先做仅开发环境可用的本地事件日志。
- 准备种子用户反馈清单：规划是否成功、地点是否可信、验证是否清晰、档案隐私是否被理解、是否愿意复用。

验收：
- 新用户无需解释即可完成主闭环。
- 验证动作能产生有用的社区内容，并更新地点可信信号。
- 应用具备可信的桌面体验，而不只是手机框中的 demo。
- 团队能判断用户是否理解并信任这个产品。

## 数据边界

项目需要清晰区分三类数据：

1. Demo 数据：内置地点、内置社区帖子、Mock AI 行为。
2. 本地用户数据：宠物档案、设置、聊天历史、保存计划、本地验证草稿。
3. 产品数据：共享地点、验证事件、社区帖子、审核后的规则更新。

阶段 1 和阶段 2 可以保持完全本地化。阶段 3 需要引入产品数据和 AI gateway 的边界，即使首版实现仍然是轻量服务。

## 架构方向

前端：
- 保留 Vite、React、TypeScript、Tailwind、React Router、Zustand、Leaflet、Lucide。
- 优先增量重构，不做整站重写。
- 保留现有 UI 组件，但逐步区分设计系统基础组件和业务组件。

状态：
- 将单个大 store 替换为领域 slice 和窄 selector。
- 持久化逻辑保持显式，并补 migration 测试。
- Mock seed data 应作为 fixture，而不是运行时 store 逻辑。

AI：
- 保留 `mockAiEngine` 作为确定性 fallback。
- 拆分 prompt 构造、provider transport、响应归一化、UI 渲染。
- 在结构化 AI 输出写入 chat state 前做 schema 校验。

测试：
- 在更好的测试覆盖相同风险前，保留当前回归脚本。
- 逐步增加标准测试结构，不一次性替换全部覆盖。
- 浏览器烟测应覆盖真实工作流，而不只做源码文本断言。

## 风险

1. 后端范围膨胀：账号、审核、同步过早进入会压垮 MVP。
2. 数据可信度：宠物友好规则很容易过期，验证记录和可信度评分比地点数量更重要。
3. 隐私信任：宠物档案进入 AI 上下文必须保持清晰、显式、可理解。
4. 包体增长：地图、图片、Mock 数据和全局 import 会悄悄增加启动成本。
5. Demo 兼容：产品化不能破坏用于路演的离线演示路径。

## 下一份实施计划

本设计确认后，第一份实施计划只覆盖阶段 1：

1. 增加 CI 质量门。
2. 修复当前 lint warning。
3. 增加一条主流程浏览器烟测。
4. 增加 bundle 体积观察。
5. 更新开发和发布命令文档。

只有阶段 1 在本地和 CI 都稳定通过后，才进入阶段 2。
